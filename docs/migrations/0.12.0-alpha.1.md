# Migration Guide: `0.12.0-alpha.1` (Planned Removals)

Status: draft (breaking changes expected)  
Last updated: 2026-02-10

This document is a **forward-looking** migration guide for the next planned
alpha cut in the fearless refactor release train.

It exists so we can:

- declare breaking removals early (while providing a deprecation window),
- keep CI + examples aligned with the recommended surface,
- avoid "surprise" API removals.

> If you are reading this before `0.12.0-alpha.1` actually lands, treat it as
> a checklist and migration preview.

---

## 1) Streaming: legacy `ChatStreamEvent` surface removed

The following legacy surfaces are planned to be **removed**:

- `ChatCapability.chatStream(...)`
- `PromptChatCapability.chatPromptStream(...)`
- `EnhancedChatCapability.chatStreamWithAdvancedTools(...)`
- `ChatStreamEvent` and its subclasses
  (`TextDeltaEvent`, `ThinkingDeltaEvent`, `ToolCallDeltaEvent`,
  `CompletionEvent`, `ErrorEvent`)

### What to use instead

Use parts-first streaming:

- Provider capability: `ChatStreamPartsCapability.chatStreamParts(...)`
- Provider capability (Prompt IR): `PromptChatStreamPartsCapability.chatPromptStreamParts(...)`
- Task-level API (recommended): `streamChatParts(...)` from `llm_dart_ai`
- Tool loop (streaming): `streamToolLoopParts(...)` from `llm_dart_ai`

### Why

`ChatStreamEvent` is a **lossy adapter** that cannot represent:

- block boundaries (`text` / `reasoning` start/end),
- citations / grounding sources,
- provider-executed tool lifecycles,
- stream-start warnings and other structural metadata.

Those are parts-only concepts in the Vercel AI SDK model.

## 2) Legacy helper aliases removed

The following helpers are planned to be removed (previously deprecated):

- all `*FromPromptIr` / `*FromPrompt` helper aliases

If you still rely on these today, migrate to the primary task APIs exported by:

```dart
import 'package:llm_dart_ai/llm_dart_ai.dart';
```

---

## 3) Recommended migration checklist

1) Replace `provider.chatStream(...)` usage with parts-first streaming:
   - provider-level: `provider.chatStreamParts(...)`
   - task-level: `streamChatParts(...)` (`llm_dart_ai`)
2) If you need citations or provider tool visibility, consume `LLMStreamPart`
   and handle:
   - `LLMSourceUrlPart` / `LLMSourceDocumentPart`
   - `LLMProviderTool*Part` (provider-executed tools)
3) Update any exhaustive `switch` on `LLMStreamPart` to include new part types
   (or use a `default` case).

## 4) Provider metadata: Google Vertex key alignment

Google Vertex provider metadata is emitted under:

- `providerMetadata['google-vertex']` (canonical)
- `providerMetadata['google-vertex.chat']` (compatibility alias)

If you previously read `providerMetadata['vertex']`, update it to
`providerMetadata['google-vertex']`.

## 5) Prompt IR: file references (`FileUrlPart` / `FileIdPart`)

Prompt IR now supports **explicit file reference parts** in addition to inline
file bytes (`FilePart`):

- `FileUrlPart(mime: ..., url: ...)`: reference a file by URL (`http(s)://`,
  `gs://...`, etc. depending on provider)
- `FileIdPart(mime: ..., id: ...)`: reference a provider-managed file id
  (e.g. Google `files/...`, OpenAI `file-...`)

### Why

This aligns with the Vercel AI SDK prompt model where "file" parts can be either:

- inline bytes (data),
- URLs,
- provider-native file ids.

It also avoids overloading `ChatMessage.protocolPayloads` as a user-facing escape
hatch for multimodal/file inputs.

### Migration impact

- If you have an **exhaustive** `switch` over `PromptPart`, update it to include
  `FileUrlPart` and `FileIdPart` (or add a `default`).
- `Prompt.toChatMessages()` cannot losslessly represent these parts and will
  throw if a prompt contains them. Use prompt-native provider capabilities:
  - `PromptChatCapability.chatPrompt(...)`
  - `PromptChatStreamPartsCapability.chatPromptStreamParts(...)`

### Provider support (high level)

- Google / Google Vertex:
  - `FileUrlPart` -> `fileData.fileUri`
  - `FileIdPart(id: 'files/...')` -> `fileData.fileUri`
  - Optional strict validation: `providerOptions['google'|'google-vertex']['supportedFileUrlsOnly']=true`
- Anthropic-compatible:
  - `FileUrlPart(application/pdf|text/plain)` -> `document.source.type=url`
  - `FileIdPart` is not supported (use URL or inline data)
- OpenAI Responses:
  - `FileUrlPart(application/pdf)` -> `input_file.file_url`
  - `FileIdPart(application/pdf)` -> `input_file.file_id`
  - `FileIdPart(image/*)` -> `input_image.file_id`
- OpenAI Chat Completions (OpenAI-compatible protocol):
  - File ids / PDF URLs are not supported; use Responses API instead.

### Examples

Google (files API id):

```dart
final prompt = Prompt(messages: [
  const PromptMessage(
    role: ChatRole.user,
    parts: [
      FileIdPart(mime: FileMime.pdf, id: 'files/123'),
      TextPart('Summarize it.'),
    ],
  ),
]);
```

OpenAI Responses (file id):

```dart
final prompt = Prompt(messages: [
  const PromptMessage(
    role: ChatRole.user,
    parts: [
      FileIdPart(mime: FileMime.pdf, id: 'file-abc'),
      TextPart('Summarize it.'),
    ],
  ),
]);
```

URL-based PDF:

```dart
final prompt = Prompt(messages: [
  const PromptMessage(
    role: ChatRole.user,
    parts: [
      FileUrlPart(
        mime: FileMime.pdf,
        url: 'https://generativelanguage.googleapis.com/v1beta/files/123',
      ),
      TextPart('Summarize it.'),
    ],
  ),
]);
```

More copy/paste examples: `docs/examples/prompt_ir_file_references.md`.

## 6) Provider-native tools: legacy web tooling flags removed

To stay close to Vercel AI SDK semantics, provider-native tools should be
configured via `providerTools` (typed `ProviderTool` catalogs).

The following legacy/best-effort enable flags and option escape hatches are
planned to be removed (or have already been removed on the fearless refactor
branch):

### Google (Gemini / Vertex)

- Removed:
  - `providerOptions['google'|'google-vertex']['webSearchEnabled']`
  - `providerOptions['google'|'google-vertex']['webSearchToolOptions']`
  - `GoogleConfig.webSearchEnabled`
- Use instead:
  - `LLMConfig(providerTools: [GoogleProviderTools.webSearch(options: ...)])`
  - or `ProviderTool(id: 'google.google_search', options: {...})`

### Anthropic

- Removed:
  - `providerOptions['anthropic'|'<anthropic-compatible>']['webFetchEnabled']`
  - `providerOptions['anthropic'|'<anthropic-compatible>']['webFetch']`
- Use instead:
  - `LLMConfig(providerTools: [AnthropicProviderTools.webFetch(toolType: ..., options: ...)])`
  - or `ProviderTool(id: 'anthropic.web_fetch_20250910', options: {...})`

### OpenAI (Responses built-in tools)

- Removed:
  - `providerOptions['openai']['webSearchEnabled']` / `fileSearchEnabled` /
    `computerUseEnabled` (legacy best-effort)
- Use instead:
  - `LLMConfig(providerTools: [...])` (recommended), and/or
  - `providerOptions['openai']['builtInTools']` (provider-specific escape hatch)
