# RBF-060 — Provider Options Policy

Date: 2026-05-27

## Outcome

Completed RBF-060.

## Reference

The Vercel AI SDK keeps provider options as a provider-keyed bag, for example
`providerOptions.openai` parsed by the OpenAI provider. The Dart library keeps
typed options as the primary local Interface, so the useful adaptation is not to
replace typed options with maps; it is to make typed provider options own their
provider namespace projection for JSON/transport seams.

## Deletion Test

Deleting typed provider-option bag projection would force HTTP transports and
cross-process callers to either drop provider-native options or duplicate the
OpenAI-family namespace encoding helpers. The new projection earns its keep
because the same typed options now define both the Dart Interface and the bag
projection.

Deleting route-local request policy was rejected. Chat Completions and
Responses compatibility warnings are tied to request body shape, route-specific
model capabilities, and warning text. Moving those rules into a generic options
policy would make the Interface broader without improving locality.

## Changes

- Added `openai_provider_options_namespaces.dart` for explicit OpenAI-family
  namespace constants.
- Made OpenAI-family language, profile-specific, embedding, image, speech, and
  transcription options implement `ProviderInvocationOptionsBagProjection`.
- Kept legacy `openAI*OptionsToProviderOptionsBag(...)` functions as
  compatibility wrappers that delegate to typed options.
- Preserved empty projection semantics in `providerOptionsBagFromInvocationOptions`.
- Added tests for common OpenAI, DeepSeek, OpenRouter, non-text projection, and
  model-kind rejection.
- Updated two OpenAI tests whose expected error text now comes from the app
  request seam introduced in RBF-040.
