# 0.11 SDK-Aligned Migration Guide

This guide matches the current SDK-aligned breaking line after the provider and
runtime package boundary refactor. It is the release-facing guide for moving
application code away from the old root compatibility surface and for updating
custom provider adapters.

The core direction is:

- build concrete models with short provider factories, such as
  `openai(...).chatModel(...)`
- call models through shared runtime helpers, usually `generateTextCall(...)`,
  `streamTextCall(...)`, `generateText(...)`, or `streamText(...)`
- reserve `doGenerate(...)` and `doStream(...)` for provider implementations and
  low-level adapters
- put request customization in `GenerateTextOptions`, `CallOptions`, or typed
  provider options
- use `ProviderMetadata` for response observations, raw provider details, and
  replay hints, not as a request option bag
- delete or rewrite old builder-era root code; the root legacy barrel and root
  provider/model/builder subpaths are no longer public migration targets

## Quick Decision Table

| Code you are migrating | Use this target | Notes |
| --- | --- | --- |
| Normal app text generation | `generateTextCall(...)` | App-facing result facade with text, usage, metadata, and optional parsed output. |
| Normal app streaming | `streamTextCall(...)` | App-facing stream facade with final result futures. |
| Raw runtime text generation | `generateText(...)` / `streamText(...)` | Runtime helpers returning raw final result or raw full-stream event shapes. |
| Step/run observation | `runTextGeneration(...)` / `streamTextRun(...)` | Advanced runtime facades for tool-loop tracing, `stepStream`, and full run results. |
| Provider implementation | `LanguageModel.doGenerate(...)` / `doStream(...)` | Implement these methods; do not import app runtime layers into provider packages. |
| Old builder code | Focused provider factories plus typed options | `LLMBuilder`, `createProvider`, and root provider/model/builder subpaths have been removed. |
| Flutter rendering | `llm_dart_flutter` plus shared `ChatMessageMapper` | Provider packages no longer own UI mappers. |
| Provider-native request features | typed provider options through `CallOptions.providerOptions` | Example: `OpenAIGenerateTextOptions`, `AnthropicGenerateTextOptions`, `GoogleGenerateTextOptions`. |
| Provider raw response or replay data | `ProviderMetadata` | Output-side data only, except replay metadata originally produced by a provider result. |
| Provider file IDs | `FileProviderReferenceData(ProviderReference...)` | Do not encode request file identity in `ProviderMetadata`. |
| Dynamic model lookup | `ProviderRegistry` with provider facades | Prefer provider-object lookup over the legacy factory-map `ModelRegistry`. |

## Imports And Dependencies

Most applications can keep depending on the root package:

```dart
import 'package:llm_dart/llm_dart.dart' as llm;
import 'package:llm_dart/core.dart' as core;
```

Use focused packages when you want smaller dependency surfaces or provider-owned
types without the root facade:

```dart
import 'package:llm_dart_ai/llm_dart_ai.dart' as ai;
import 'package:llm_dart_openai/llm_dart_openai.dart' as openai;
```

The main package roles are:

| Package or entrypoint | Role |
| --- | --- |
| `package:llm_dart/llm_dart.dart` | Modern root facade, provider factories, and shared runtime exports. |
| `package:llm_dart/core.dart` | Shared model contracts, runtime helpers, serialization, and UI projection from the root package. |
| `llm_dart_ai` | Framework-neutral runtime helpers, structured output, stream runners, and UI mapping. |
| `llm_dart_provider` | Provider contracts, prompt/result/event data structures, provider options, and metadata. |
| `llm_dart_openai`, `llm_dart_anthropic`, `llm_dart_google` | Provider-owned model factories, codecs, typed options, native tools, and provider-specific helpers. |
| `llm_dart_ollama`, `llm_dart_elevenlabs` | Dedicated provider packages for local/community-style provider surfaces. |
| `llm_dart_chat` | Framework-neutral chat/session layer. |
| `llm_dart_flutter` | Flutter adapters and widgets. |
| `llm_dart_transport` | Shared transport abstractions and Dio transport implementation. |

Historical references to `llm_dart_community` should be migrated to
`llm_dart_ollama` or `llm_dart_elevenlabs`.

For dynamic model lookup, register provider facades with `ProviderRegistry`:

```dart
final registry = llm.ProviderRegistry(
  providers: {
    'openai': llm.openai(apiKey: openAIKey),
    'google': llm.google(apiKey: googleKey),
    'ollama': llm.ollama(),
  },
);

final model = registry.languageModel('openai:gpt-4.1-mini');
```

`ModelRegistry` is now a low-level compatibility factory registry. Keep it only
for custom adapters that already own independent per-kind factories; new
application code should use provider-object registration instead.

`ProviderRegistry` respects provider-declared model facet support. OpenAI-family
compatible providers such as OpenRouter, DeepSeek, Groq, xAI, and Phind are
registered as language-model providers through the shared OpenAI-family facade,
but they are not listed as embedding, image, speech, or transcription providers
unless their profile explicitly supports that model kind. Unsupported lookups
such as `registry.embeddingModel('openrouter:text-embedding-3-small')` now fail
with `UnsupportedError`.

## 1. Replace `ai()` And `LLMBuilder()` With Focused Factories

`ai()` and the root `LLMBuilder` compatibility surface have been removed. New
and migrated code should construct concrete models from focused provider
facades.

Before:

```dart
final provider = await ai()
    .openai()
    .apiKey(openAIKey)
    .model('gpt-4.1-mini')
    .build();
```

After:

```dart
import 'package:llm_dart/llm_dart.dart' as llm;
import 'package:llm_dart/core.dart' as core;

final model = llm.openai(apiKey: openAIKey).chatModel('gpt-4.1-mini');

final result = await core.generateTextCall(
  model: model,
  messages: [
    core.UserModelMessage.text('Explain Dart in one sentence.'),
  ],
);

print(result.text);
```

Use focused factories as the migration target. New code should usually
start from short provider factories such as `openai(...)`, `google(...)`,
`anthropic(...)`, `ollama(...)`, and `elevenLabs(...)`.

## 2. Replace Legacy Chat Calls

Old compatibility providers usually exposed chat-like methods directly on the
provider object.

Before:

```dart
final response = await provider.chat([
  ChatMessage.user('Explain Dart in one sentence.'),
]);

print(response.text);
```

After:

```dart
final result = await core.generateTextCall(
  model: model,
  messages: [
    core.UserModelMessage.text('Explain Dart in one sentence.'),
  ],
);

print(result.text);
```

App-facing message migration for the common subset is mechanical:

| Old message | New message |
| --- | --- |
| `ChatMessage.system(text)` | `SystemModelMessage.text(text)` |
| `ChatMessage.user(text)` | `UserModelMessage.text(text)` |
| `ChatMessage.assistant(text)` | `AssistantModelMessage.text(text)` |
| legacy tool-result blocks | provider-contract `ToolPromptMessage` with `ToolResultPromptPart` when replaying normalized provider prompts |

Use `PromptMessage` types directly only at the provider-contract layer or when
replaying already-normalized provider prompts.

Provider-native replay details that do not fit the shared prompt shape should
stay provider-owned and travel as provider metadata produced by previous
provider results.

## 3. Replace Legacy Streaming Calls

Before:

```dart
await for (final event in provider.chatStream(messages)) {
  // Handle old compatibility stream events.
}
```

After:

```dart
final stream = core.streamTextCall(
  model: model,
  messages: [
    core.UserModelMessage.text('Stream a concise answer.'),
  ],
);

await for (final event in stream) {
  switch (event) {
    case core.TextDeltaEvent(:final delta):
      stdout.write(delta);
    case core.ReasoningDeltaEvent(:final delta):
      stderr.write(delta);
    case core.FinishEvent(:final finishReason, :final usage):
      stdout.writeln('\n$finishReason tokens=${usage?.totalTokens}');
    default:
      break;
  }
}

final result = await stream.result;
print(result.text);
```

For tool loops or multi-step flows that need direct step observation, use the
advanced runtime runner facade:

```dart
final run = core.streamTextRun(
  model: model,
  messages: [
    core.UserModelMessage.text('Use tools if needed.'),
  ],
  tools: tools,
  functionToolExecutor: executor,
);

await for (final event in run.eventStream) {
  // Render or observe shared stream events.
}

final finalResult = await run.result;
```

## 4. Update Direct `LanguageModel` Calls

The `LanguageModel` contract is now implementation-facing. Old direct
`model.generate(request)` and `model.stream(request)` calls have been replaced
with `doGenerate(request)` and `doStream(request)`.

For app code, prefer translating the request into runtime helper arguments:

```dart
final result = await core.generateTextCall(
  model: model,
  prompt: request.prompt,
  tools: request.tools,
  toolChoice: request.toolChoice,
  options: request.options,
  callOptions: request.callOptions,
);
```

For app code that intentionally wants the raw runtime result instead of the
text-call facade:

```dart
final result = await core.generateText(
  model: model,
  prompt: request.prompt,
  tools: request.tools,
  toolChoice: request.toolChoice,
  options: request.options,
  callOptions: request.callOptions,
);
```

For provider adapters and tests that intentionally exercise the provider
contract:

```dart
final result = await model.doGenerate(request);
final events = model.doStream(request);
```

Custom provider implementations should implement the `do*` methods:

```dart
final class MyLanguageModel implements core.LanguageModel {
  @override
  Future<core.GenerateTextResult> doGenerate(
    core.GenerateTextRequest request,
  ) async {
    // Provider wire call and result decoding.
    throw UnimplementedError();
  }

  @override
  Stream<core.LanguageModelStreamEvent> doStream(
    core.GenerateTextRequest request,
  ) async* {
    // Provider wire stream and event decoding.
  }
}
```

Provider packages should not depend on `llm_dart_ai`, `llm_dart_chat`,
`llm_dart_flutter`, or the root `llm_dart` package at runtime.

## 5. Migrate Stream Layers Deliberately

There are now three separate stream layers. Do not substitute one for another:

| Layer | Type | Owner |
| --- | --- | --- |
| Provider model call | `LanguageModelStreamEvent` | `llm_dart_provider` and focused provider packages |
| AI runtime run | `TextStreamEvent` | `llm_dart_ai` / root `core.dart` |
| Chat/session UI | `ChatUiStreamChunk` | `llm_dart_ai`, re-exported by `llm_dart_chat` |

Provider adapters should emit only provider model-call events such as
`StartEvent`, `TextDeltaEvent`, `ToolInputDeltaEvent`, and `FinishEvent`.
Runtime-only events such as `RunStartEvent`, `RunFinishEvent`,
`StepStartEvent`, `StepFinishEvent`, and `AbortEvent` are emitted by
`streamText(...)`, not by provider packages.

For in-process chat/session streaming, consume `Stream<ChatUiStreamChunk>` and
`ChatUiStreamReader`. For HTTP/SSE transport implementations, use
`HttpChatTransportChunk`, `HttpChatTransportChunkJsonCodec`, and
`HttpChatTransportServerAdapter`; those wire chunks are not model/runtime
events.

## 6. Move Request Configuration Out Of `ProviderMetadata`

`ProviderMetadata` is output-side by default. If a value changes what the
provider sends, samples, stores, caches, searches, thinks, or executes, it is
request configuration and belongs elsewhere.

Use shared `GenerateTextOptions` for durable cross-provider controls:

```dart
final result = await core.generateTextCall(
  model: model,
  messages: [
    core.UserModelMessage.text('Write a concise product summary.'),
  ],
  options: const core.GenerateTextOptions(
    temperature: 0.4,
    maxOutputTokens: 300,
    presencePenalty: 0.1,
    frequencyPenalty: 0.1,
    seed: 42,
    reasoning: core.GenerateTextReasoningOptions.enabled(
      effort: core.ReasoningEffort.low,
    ),
    includeRawChunks: false,
  ),
);
```

Use `CallOptions` for invocation controls:

```dart
final result = await core.generateTextCall(
  model: model,
  prompt: prompt,
  callOptions: const core.CallOptions(
    timeout: Duration(seconds: 30),
    maxRetries: 2,
    headers: {
      'x-request-source': 'migration-guide',
    },
  ),
);
```

Use typed provider options for provider-native request features:

```dart
final result = await core.generateTextCall(
  model: model,
  prompt: prompt,
  callOptions: const core.CallOptions(
    providerOptions: llm.OpenAIGenerateTextOptions(
      metadata: {
        'ticket_id': 'T-123',
      },
      previousResponseId: 'resp_123',
      promptCacheKey: 'ticket-summary',
    ),
  ),
);
```

Other provider examples:

```dart
final anthropicResult = await core.generateTextCall(
  model: anthropicModel,
  prompt: prompt,
  callOptions: const core.CallOptions(
    providerOptions: llm.AnthropicGenerateTextOptions(
      metadata: {
        'ticket_id': 'T-123',
      },
      container: 'support-bot',
      extendedThinking: true,
      thinkingBudgetTokens: 1024,
    ),
  ),
);

final googleResult = await core.generateTextCall(
  model: googleModel,
  prompt: prompt,
  callOptions: const core.CallOptions(
    providerOptions: llm.GoogleGenerateTextOptions(
      includeThoughts: true,
      thinkingBudgetTokens: 1024,
    ),
  ),
);
```

The exception for `ProviderMetadata` is replay: prompt, content, or tool parts
may carry provider metadata when that metadata came from an earlier provider
result and is required to continue the conversation faithfully.

### Replace `CompatWebSearchPresets`

`CompatWebSearchPresets` has been removed from the legacy barrel. New code
should use provider-owned typed search options instead of shared presets.

For compatibility code that still needs to preserve the old OpenRouter
online-model bridge, construct the compatibility config explicitly:

```dart
const webSearchConfig = WebSearchConfig(
  maxResults: 5,
  searchPrompt: 'Focus on recent developments.',
  strategy: WebSearchStrategy.plugin,
  searchType: WebSearchType.web,
);
```

For modern code, prefer provider-owned options such as
`OpenRouterChatModelSettings(search: OpenRouterSearchOptions.onlineModel())`,
`OpenAIGenerateTextOptions(builtInTools: [OpenAIWebSearchTool()])`,
`AnthropicGenerateTextOptions(tools: [...])`, or `XAIGenerateTextOptions`.

## 7. Move File IDs To `ProviderReference`

Provider-specific file identity is no longer request metadata.

Before:

```dart
core.FilePromptPart(
  mediaType: 'application/pdf',
  data: core.FileUrlData(Uri.parse('https://example.test/doc.pdf')),
  providerMetadata: core.ProviderMetadata.forNamespace('openai', {
    'fileId': 'file_123',
  }),
);
```

After:

```dart
core.FilePromptPart(
  mediaType: 'application/pdf',
  data: core.FileProviderReferenceData(
    core.ProviderReference.forProvider('openai', 'file_123'),
  ),
);
```

Use `ProviderMetadata` on file or content parts only for output-side replay data
that was produced by a provider response.

## 8. Replace Provider UI Mappers With Shared Mapping

Provider packages no longer own Flutter or UI mapper layers. Use the shared
mapper as the baseline and keep provider-specific rendering decisions in the app
or UI layer.

```dart
final mapped = const core.ChatMessageMapper().map(message);

print(mapped.text);
print(mapped.reasoningText);
print(mapped.responseProviderMetadata?.namespace('openai'));

for (final part in mapped.customParts) {
  print('${part.kind}: ${part.data}');
}
```

When rendering direct generation results rather than chat messages, inspect
shared content parts and provider metadata explicitly:

```dart
for (final part in result.content) {
  switch (part) {
    case core.TextContentPart(:final text):
      print(text);
    case core.CustomContentPart(:final kind, :final data):
      print('$kind: $data');
    default:
      break;
  }
}

print(result.providerMetadata?.namespace('openai'));
```

## 9. Use Provider Packages For Provider-Native Surfaces

Provider-native helpers stay in their provider packages instead of widening the
shared runtime surface:

```dart
import 'package:llm_dart_openai/llm_dart_openai.dart' as openai;

final client = openai.openai(apiKey: openAIKey);
final model = client.chatModel('gpt-4.1-mini');
final files = client.files();
```

Ollama and ElevenLabs are dedicated packages now:

```dart
import 'package:llm_dart_ollama/llm_dart_ollama.dart' as ollama;
import 'package:llm_dart_elevenlabs/llm_dart_elevenlabs.dart' as elevenlabs;

final localModel = ollama.ollama().chatModel('llama3.2');
final speechModel = elevenlabs.elevenLabs(apiKey: elevenLabsKey)
    .speechModel('eleven_multilingual_v2');
```

Keep residual legacy root provider shells only for migration code that still
needs compatibility-era capability interfaces or provider-specific APIs that do
not yet have a modern provider-package equivalent.

## 10. Compatibility Checklist

Before considering a migration complete, verify these points:

- no executable code calls `ai()`
- app text calls use `generateTextCall(...)`, `streamTextCall(...)`, or
  the raw runtime helpers `generateText(...)` / `streamText(...)`; use
  `runTextGeneration(...)` / `streamTextRun(...)` only when direct step/run
  observation is needed
- custom providers implement `doGenerate(...)` and `doStream(...)`
- custom provider streams return `LanguageModelStreamEvent`; app/runtime
  streams consume `TextStreamEvent`
- chat/session stream code consumes `ChatUiStreamChunk`; HTTP transport code
  consumes `HttpChatTransportChunk`
- provider packages do not import root `llm_dart`, `llm_dart_ai`,
  `llm_dart_chat`, or `llm_dart_flutter` at runtime
- request-side provider features use typed provider options
- response-side provider details use `ProviderMetadata`
- provider file IDs use `FileProviderReferenceData` and `ProviderReference`
- UI code imports `ChatMessageMapper` from the shared runtime/chat/Flutter layer,
  not from provider packages
- old `llm_dart_community` imports have moved to `llm_dart_ollama` or
  `llm_dart_elevenlabs`
- no executable code imports `package:llm_dart/legacy.dart`,
  `package:llm_dart/builder/...`, `package:llm_dart/models/...`,
  `package:llm_dart/providers/...`, or legacy `package:llm_dart/core/...`
  subpaths
